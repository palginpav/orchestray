**Section 22a: Automatic Pattern Extraction (Post-Orchestration)** — per D-01, D-02, D-03, D-04, D-05:

After Section 15 step 3 completes (audit trail archived to `.orchestray/history/<orch-id>/`, cleanup done), extract reusable patterns:

1. Read `.orchestray/history/<orch-id>/events.jsonl` from the just-archived orchestration.
2. Also read `.orchestray/history/<orch-id>/state/task-graph.md` if it exists (for decomposition context).
3. Identify extractable patterns across four categories (per D-02):
   - **decomposition:** Task breakdown strategies that led to success (zero re-plans, zero verify-fix failures). Record the decomposition approach from the task graph.
   - **routing:** Model routing decisions that proved correct — `routing_outcome` events where the chosen model completed without escalation.
   - **specialization:** Dynamic agents saved as specialists (`specialist_saved` events) or specialist reuses that succeeded.
   - **anti-pattern:** Re-plan triggers (`replan` events), verify-fix failures (`verify_fix_fail`), escalations (`escalation` events). Record what went wrong and why. Per D-03, learn from failures too.
4. Skip extraction when: orchestration was simple (2-3 tasks, standard architect->developer->reviewer flow with no novel insight), OR an equivalent pattern already exists in `.orchestray/patterns/` with higher confidence (update the existing pattern's Evidence section instead).
5. Before writing a new pattern, check if `.orchestray/patterns/` already has a substantially similar pattern. Update existing rather than duplicate.
6. Write pattern files to `.orchestray/patterns/{category}-{name}.md` (per D-04) with this template (per D-05):

```markdown
---
name: {kebab-case-name}
category: {decomposition|routing|specialization|anti-pattern}
confidence: {0.5 for positive patterns, 0.6 for anti-patterns}
times_applied: 0
last_applied: null
created_from: {orch-id}
description: {one-line description for matching}
---

# Pattern: {Human Readable Name}

## Context
{When this pattern applies — task type, domain, characteristics}

## Approach
{What to do (positive) or what to avoid (anti-pattern)}

## Evidence
- {orch-id}: {brief outcome description}
```

7. Report to user: show a brief table of extracted patterns (Name, Category, Confidence). If no patterns extracted, say "No novel patterns identified from this orchestration."

**Section 22c: Confidence Feedback Loop** — per D-09:

After orchestration completes but BEFORE extracting new patterns (Section 22a), update confidence for any patterns that were applied during this orchestration:

1. For each pattern noted as "applied" during Section 22b:
   - If orchestration status = "success": increase confidence by +0.1 (cap at 1.0)
   - If orchestration status = "partial": no change (+0.0)
   - If orchestration status = "failure": decrease confidence by -0.2 (floor at 0.0)
   - Increment `times_applied` by 1
   - Set `last_applied` to current ISO 8601 timestamp
2. Write updated frontmatter back to the pattern file.

**Section 22d: Pruning** — per D-06:

After writing new patterns in Section 22a step 6:

1. Count all `.md` files in `.orchestray/patterns/`.
2. If count > 50: compute `score = confidence * times_applied` for each pattern.
3. Sort ascending. Remove patterns with the lowest scores until count = 50.
4. Log: "Pruned {N} low-value patterns: {names}"
5. Append `pattern_pruned` event(s) to the current audit trail (if still active) or note in output.

Also add cross-references in Section 15 Step 3. After step 9 (cost summary), add:

```
10. **Update applied pattern confidence** per Section 22c (if any patterns were applied during this orchestration).

11. **Extract new patterns** per Section 22a from the just-archived history at `.orchestray/history/<orch-id>/events.jsonl`.
```

The ordering in Section 15 step 3 is critical (per research Pitfall 1): archival (step 7) -> cleanup (step 8) -> cost report (step 9) -> confidence feedback (step 10) -> extraction (step 11). Extraction reads from the ARCHIVED history directory, not the active audit directory.

## Trust Boundaries

| Pattern files -> PM reasoning | PM reads `.orchestray/patterns/` files that could be user-edited |
| Audit trail -> Pattern extraction | PM extracts patterns from events.jsonl which is generated by the system |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Tampering | `.orchestray/patterns/*.md` | accept | Patterns are advisory only (D-08). Tampered patterns cannot force PM behavior, only influence reasoning. User has full edit access to their own project files. Low risk. |
| T-07-02 | Denial of Service | Pattern pruning | mitigate | Cap at 50 patterns (D-06). Pruning runs after every extraction. PM globs directory; if >50 files exist, prune before proceeding. Prevents unbounded growth. |
| T-07-03 | Information Disclosure | Pattern files contain orchestration details | accept | Patterns are stored in `.orchestray/` which is gitignored. They contain abstracted strategies, not secrets. Users control their own project directory. |
| T-07-04 | Elevation of Privilege | Pattern content could include instructions to PM | accept | Patterns are read as data (frontmatter + markdown). PM treats them as advisory context for reasoning, same as specialist descriptions (Section 21). No tool grants or permission changes possible through pattern content. |
