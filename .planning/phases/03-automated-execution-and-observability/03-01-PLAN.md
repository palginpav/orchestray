**Part A: Add Section 14 to PM agent (agents/pm.md)**

Add a new "## 14. Parallel Execution Protocol" section at the end of agents/pm.md (after Section 13). This section implements D-03 and D-04.

The section must contain:

1. **When to use:** When task decomposition (Section 13) identifies parallel groups — multiple tasks in the same group with no inter-dependencies.

2. **Spawning parallel agents (D-03 dual isolation):**
   - For each task in a parallel group, spawn the assigned agent
   - Each agent runs with worktree isolation (the `isolation: worktree` frontmatter field handles this)
   - File ownership is already assigned in Section 13 decomposition — this is the first isolation layer
   - Worktree isolation is the safety net — each agent's changes are on a separate branch
   - Worktree branch naming: `orchestray/<orch-id>/task-<N>` (e.g., `orchestray/orch-1712345678/task-3`)
   - PM spawns agents for all tasks in the group, then waits for all to complete

3. **Sequential merge after completion (D-04):**
   - After ALL agents in the parallel group have completed:
   - For each completed agent's worktree (in task order):
     a. Review the diff: run `git diff main...<worktree-branch>` to see changes
     b. Verify file ownership: check that changes are ONLY in files assigned to that task (from task graph "Files (write)" field)
     c. If ownership violated: log the violation to `.orchestray/audit/events.jsonl` as a `file_ownership_violation` event (JSON with fields: timestamp, type, orchestration_id, task_id, agent_type, file, assigned_files). Continue with merge but warn user.
     d. Merge: run `git merge <worktree-branch>` into the current branch
     e. If merge conflict: attempt resolution. If unresolvable, keep first-merged version and log the conflict.
     f. Clean up: run `git worktree remove <path>` after successful merge
   - After all worktrees merged: run tests/lint if applicable
   - Update state: mark the parallel group as complete in `.orchestray/state/task-graph.md`

4. **Waiting behavior:**
   - While waiting for parallel agents, inform user: "Orchestration in progress: {N} agents working in parallel. Waiting for completion..."
   - Do NOT spawn additional work during the parallel wait
   - Do NOT process new user prompts as orchestration tasks during the wait

5. **Error handling:**
   - If an agent fails: log failure, continue with other agents in the group, report failed task to user after group completes
   - If merge fails: log conflict details, attempt manual resolution, fall back to sequential re-execution of the conflicting task if needed

6. **Integration with task graph execution flow:**
   - After Section 13 produces the task graph, execution proceeds group by group:
     - Group 1 (roots): Execute via this parallel protocol
     - Collect results, update state, perform context handoffs (Section 11)
     - Group 2: Execute via this parallel protocol
     - Continue until all groups complete

| Worktree merge -> main branch | Changes from agent worktrees merge into the user's working branch |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-03-01 | Tampering | Worktree merge | mitigate | PM verifies file ownership before merge — changes outside assigned files are logged as violations and reported to user |
| T-03-02 | Denial of Service | Parallel agent spawning | accept | maxTurns limits prevent runaway agents. PM caps subtasks at 6 per Section 13. Low risk in local CLI context. |
| T-03-03 | Information Disclosure | Agent worktree branches | mitigate | PM cleans up worktrees after merge via `git worktree remove`. No stale branches with potentially sensitive changes. |
