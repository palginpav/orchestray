**`skills/run/SKILL.md`** -- The primary orchestration trigger (D-08):

```markdown
---
name: run
description: Trigger multi-agent orchestration on a task
disable-model-invocation: true
argument-hint: [task description]
---

# Orchestrate Task

You are receiving this because the user invoked `/orchestray:run`. Orchestrate the following task using your multi-agent delegation protocol.

## Task

$ARGUMENTS

## Orchestration Instructions

Follow your Task Assessment Protocol to evaluate this task:

1. **Assess complexity**: Classify the task as Simple, Medium, or Complex per your assessment criteria.

2. **If Simple**: Handle the task directly without spawning subagents. Explain to the user that orchestration is not needed for this task and proceed to complete it.

3. **If Medium or Complex**: Execute your Delegation Strategy:
   a. Initialize orchestration state: create `.orchestray/` directory if it does not exist, write `current-task.json` with task description and status "in_progress"
   b. Decompose the task into subtasks
   c. Determine delegation pattern (sequential, parallel, or selective) per your Directed Fan-Out protocol
   d. Spawn specialist agents for each subtask
   e. Collect and validate results from each agent
   f. Handle any failures using your Retry Protocol
   g. Update orchestration state as agents complete
   h. Report final results to the user

4. **Always**: Tell the user what you decided (orchestrate or solo) and why.

## Output

After orchestration completes:
- Summarize what each agent did and the outcome
- List all files changed across all agents
- Report any issues or warnings from agents
- Move orchestration state to `.orchestray/history/{timestamp}.json`
```

**`skills/config/SKILL.md`** -- Configuration management (D-08):

```markdown
---
name: config
description: View or modify orchestration settings
disable-model-invocation: true
argument-hint: [setting] [value] or empty to show all
---

# Orchestration Configuration

The user wants to view or modify orchestration settings.

## Configuration Protocol

1. **Parse arguments**: `$ARGUMENTS`
   - If empty: Show all current settings
   - If one argument: Show that specific setting's current value
   - If two arguments: Set the first to the value of the second

2. **Configuration file**: Settings are stored in `.orchestray/config.json`. If it does not exist, use these defaults:

```json
{
  "auto_review": true,
  "max_retries": 1,
  "default_delegation": "sequential",
  "verbose": false
}
```

### Available Settings

| Setting | Type | Default | Description |
|---------|------|---------|-------------|
| `auto_review` | boolean | `true` | Automatically run reviewer after developer completes |
| `max_retries` | number | `1` | Maximum retry attempts on agent failure (0-2) |
| `default_delegation` | string | `"sequential"` | Default delegation pattern: "sequential", "parallel", or "selective" |
| `verbose` | boolean | `false` | Show detailed agent progress during orchestration |

3. **Validation**: 
   - `max_retries` must be 0, 1, or 2
   - `default_delegation` must be one of: "sequential", "parallel", "selective"
   - `auto_review` and `verbose` must be boolean (true/false)
   - Reject invalid values with a helpful error message

4. **Output format**:

When showing settings:
```
## Orchestray Configuration

| Setting | Value | Description |
|---------|-------|-------------|
| auto_review | true | Automatically run reviewer after developer |
| max_retries | 1 | Maximum retry attempts on agent failure |
| default_delegation | sequential | Default delegation pattern |
| verbose | false | Show detailed agent progress |

Use `/orchestray:config [setting] [value]` to change a setting.
```

When setting a value:
```
Updated `{setting}` from `{old_value}` to `{new_value}`.

**Note:** In Phase 1, the config skill stores and retrieves settings in `.orchestray/config.json` but the PM agent does not yet read these values to modify its behavior. PM integration with config values (e.g., honoring `auto_review`, `max_retries`, `default_delegation`, `verbose`) is planned for Phase 2.
```
```

# Orchestration Report

The user wants an audit report of a completed orchestration.

## Report Generation Protocol

1. **Find the orchestration**: 
   - If `$ARGUMENTS` is empty: Use the most recent file in `.orchestray/history/`
   - If `$ARGUMENTS` is provided: Find the matching orchestration in `.orchestray/history/`
   - If no history exists: Report "No completed orchestrations found. Run `/orchestray:run [task]` first."

2. **Read the orchestration record**: Parse the JSON file from history.

3. **Generate the report** in this format:

```
## Orchestration Report

**Task:** [original task description]
**Started:** [timestamp]
**Completed:** [timestamp]
**Duration:** [elapsed time]
**Status:** [success/partial/failure]

### Agent Activity

| Agent | Task | Status | Files Changed |
|-------|------|--------|---------------|
| architect | [subtask] | success | [files] |
| developer | [subtask] | success | [files] |
| reviewer | [review] | success | - |

### Files Changed (All Agents)
- [deduplicated list of all files changed across all agents]

### Issues Found
| Severity | Agent | Description |
|----------|-------|-------------|
| warning | reviewer | [issue description] |

### Recommendations
- [aggregated recommendations from all agents]

### Agent Results

#### Architect
[architect's result summary]

#### Developer
[developer's result summary]

#### Reviewer
[reviewer's result summary]
```

4. **Edge cases**:
   - If an orchestration was handled solo (no agents spawned), report that: "This task was handled directly without orchestration."
   - If orchestration failed, include failure details and retry attempts.
```
