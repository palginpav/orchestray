   - Task was one-off / unlikely to recur
   - An existing specialist already covers this domain (update description instead if needed)
   - The dynamic agent prompt is too task-specific to generalize

3. **Save Process** (numbered steps):
   1. Read `.orchestray/specialists/registry.json` (if missing: create `.orchestray/specialists/` directory and initialize `registry.json` with `{ "version": 1, "specialists": [] }`)

   6. Delete the `agents/{name}.md` copy (as before)
   7. Log `specialist_saved` event to `.orchestray/audit/events.jsonl`:
      ```json
      { "timestamp": "<ISO 8601>", "type": "specialist_saved", "orchestration_id": "<current>", "agent_name": "{name}", "source": "auto" }
      ```
   8. Report to user: "Saved '{name}' specialist for future reuse."

   - On decline: continue normally, do not ask again until next use increment.

**Add Section 21 immediately after Section 20.**

Create "## 21. Specialist Reuse Protocol" with the following content (50-80 lines):

1. **When to Check Registry:** Only when Section 17 would normally create a dynamic agent. Do NOT check on every orchestration -- only when a subtask has been identified as needing a specialist (Section 17 criteria met).

2. **Registry Check** (numbered steps):
   1. Read `.orchestray/specialists/registry.json`.
      - If file or directory missing: no specialists available, proceed to Section 17 normal flow (create new dynamic agent).
   2. **File sync for user-created specialists** (per D-10, D-08): Scan `.orchestray/specialists/` for `.md` files not in `registry.json`. For each unregistered file:
      a. **Validate** (per D-09): Read file, check YAML frontmatter contains required fields: `name` (string), `description` (string), `tools` (comma-separated string from allowed set: `Read`, `Glob`, `Grep`, `Bash`, `Write`, `Edit`). Reject files with `bypassPermissions` or `acceptEdits` fields in frontmatter (security).
      b. If valid: auto-register with `{ "name": "{from frontmatter}", "description": "{from frontmatter}", "source": "user", "file": "{filename}", "times_used": 0, "last_used": null, "created_at": "{ISO 8601 now}" }`. Write updated `registry.json`.
      c. If invalid: skip the file, log a warning internally: "Skipped invalid specialist file: {filename} -- missing required fields or invalid tool names." Do NOT crash the orchestration.
   3. **Match subtask against registry** (per D-02): Compare the subtask's description and domain against specialist `name` and `description` fields. Use reasoning to determine if a specialist is a good match for the subtask. Do NOT load full .md files during matching -- only read names and descriptions from registry.json.
      - **Priority rule** (per D-08): If both a `source: "user"` and `source: "auto"` specialist match, prefer the user-created one.
   4. **If match found:**
      a. Copy `.orchestray/specialists/{file}` to `agents/{name}.md`
      b. Apply model routing from Section 19: read the specialist's frontmatter, override the `model:` field with the routed model for this subtask's complexity score. Write the updated file.
      c. Proceed to Section 17 step 2 (spawn the agent).
      d. After completion (in Section 17 step 5): increment `times_used`, set `last_used` to current ISO 8601 timestamp in `registry.json`. Delete `agents/{name}.md` copy.
      e. Log `specialist_reused` event to `.orchestray/audit/events.jsonl`:
         ```json
         { "timestamp": "<ISO 8601>", "type": "specialist_reused", "orchestration_id": "<current>", "agent_name": "{name}", "times_used": "{new count}" }
         ```
   5. **If no match:** Proceed to Section 17 normal flow (create new dynamic agent from scratch).

3. **Selection Display:** When announcing specialist reuse, format as:
   ```
   Reusing specialist '{name}' ({model} -- score {N}/12)
   ```

4. **Staleness Warning:** If a matched specialist has `last_used` older than 30 days, note internally that the specialist may reference outdated APIs or file paths. Proceed with reuse but monitor output quality more carefully.

5. **Allowed tool names for validation:** `Read`, `Glob`, `Grep`, `Bash`, `Write`, `Edit`. Any other tool name in the `tools` frontmatter field makes the specialist file invalid.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User -> .orchestray/specialists/ | User can drop arbitrary .md files that get registered and spawned as agents |
| Specialist .md -> agents/ directory | Specialist templates copied to agents/ get full tool access specified in their frontmatter |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Elevation of Privilege | Specialist .md frontmatter | mitigate | Validation rejects `bypassPermissions` and `acceptEdits` fields in specialist frontmatter (Section 21 step 2a). Tool names validated against allowed set. |
| T-06-02 | Tampering | registry.json | accept | Single-user local plugin. Registry is in gitignored .orchestray/ directory. Risk of external tampering is minimal. |
| T-06-03 | Spoofing | Specialist name collision with core agents | mitigate | Name validation rule: specialist names must NOT be pm, architect, developer, or reviewer (Section 17 note). |
| T-06-04 | Denial of Service | Registry bloat | mitigate | Soft cap at 20 specialists with warning (D-04, Section 20). Matching reads only names+descriptions, not full .md files. |
| T-06-05 | Information Disclosure | Specialist prompts contain project-specific info | accept | Specialists are local, gitignored, single-user. No sharing mechanism exists. Low risk. |
</threat_model>
