Create the directory `skills/orchestray:specialists/` and the file `skills/orchestray:specialists/SKILL.md`.

**Frontmatter** (use standard YAML frontmatter delimited by triple-dash lines):
- name: specialists
- description: List, view, edit, and remove persistent specialists
- disable-model-invocation: true
- argument-hint: list | view {name} | remove {name} | edit {name}

**Markdown body** (following the orchestray:config skill pattern exactly):

Start with heading "# Specialist Management" and intro line "The user wants to manage persistent specialist agents."

Then "## Protocol" section with:

1. **Parse arguments**: `$ARGUMENTS`
   - If empty or "list": Show all specialists in table format
   - If starts with "view": Show full content of named specialist
   - If starts with "remove": Delete named specialist with confirmation
   - If starts with "edit": Allow user to modify named specialist

2. **Registry file**: `.orchestray/specialists/registry.json`. If directory or file does not exist, report: "No specialist registry found. Specialists are created automatically during orchestration when dynamic agents succeed, or you can place custom .md files in `.orchestray/specialists/`."

**### List Operation** (per D-12):

Read `registry.json` and display table with columns: `| Name | Source | Uses | Last Used | Description |` with separator row `|------|--------|------|-----------|-------------|`. Each specialist gets a row. Below the table show "{N} specialist(s) registered." and usage hints for view/remove/edit subcommands.

If no specialists registered, show helpful message about how specialists get created.

**### View Operation:**

Parse specialist name from arguments (second word after "view"). Look up name in `registry.json` to get `file` field. If not found, report error with suggestion to use list. If found, read `.orchestray/specialists/{file}` and display: heading "## Specialist: {name}", metadata line with Source/Uses/Last Used/Created, then "### Definition" with the full .md file content.

**### Remove Operation:**

Parse specialist name. Look up in `registry.json`. If not found, report error. If found, ask for confirmation: "Remove specialist '{name}' ({source}, used {times_used} times)? This cannot be undone. (yes/no)". On confirmation: delete the .md file from `.orchestray/specialists/`, remove entry from `registry.json` specialists array, write updated `registry.json`, report "Removed specialist '{name}'.". On decline: "Cancelled. Specialist '{name}' retained."

**### Edit Operation:**

Parse specialist name. Look up in `registry.json`. If not found, report error. If found, read `.orchestray/specialists/{file}` and display current content. Ask user what they want to change. Apply changes to the .md file. **Re-validate after edit** per D-09: YAML frontmatter must contain `name` (string), `description` (string), `tools` (comma-separated, from allowed set: `Read`, `Glob`, `Grep`, `Bash`, `Write`, `Edit`). Reject if `bypassPermissions` or `acceptEdits` fields are present. If validation fails, revert the edit, report what is wrong, ask user to fix. If validation passes, write updated file, update `description` in `registry.json` if changed. Report: "Updated specialist '{name}'."

**### No Test/Dry-Run Mode** (per D-13):

Add a note: "This skill does NOT include test or dry-run functionality. Users inspect specialists via `view` and test them naturally during orchestration."
