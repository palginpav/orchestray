**New Section: "## 12. Complexity Scoring" (per D-11, D-12, D-13)**

### Scoring Signals
Evaluate every task using these four signals, each scoring 0-3 points:

1. **File/Module Count** (estimated files needing modification):
   - 0 points: 1 file
   - 1 point: 2 files
   - 2 points: 3-5 files
   - 3 points: 6+ files

2. **Cross-Cutting Concerns** (count distinct domains: auth, DB, UI, API, tests, config, infra):
   - 0 points: 1 domain
   - 1 point: 2 domains
   - 2 points: 3 domains
   - 3 points: 4+ domains

3. **Task Description Signals**:
   - 0 points: Short, clear, single action
   - 1 point: >100 chars or minor ambiguity
   - 2 points: >200 chars or ambiguity markers ("maybe", "or", "consider")
   - 3 points: >300 chars or scope markers ("all", "entire", "across", "everything")

4. **Keyword Patterns**:
   - 0 points: "fix", "typo", "add field", "update", "rename"
   - 1 point: "add", "create", "implement" (single feature)
   - 2 points: "refactor", "redesign", "restructure"
   - 3 points: "migrate", "overhaul", "rewrite", "rebuild"

### Score Mapping
- **0-3 = Simple**: Handle solo. Do NOT orchestrate.
- **4-7 = Medium**: Orchestrate. Decompose into subtasks.
- **8-12 = Complex**: Orchestrate. Full decomposition with all agent types.

### Threshold (per D-12)
Default threshold: orchestrate at score >= 4 (medium+). Conservative -- better to under-orchestrate than over-orchestrate.

### User Override (per D-13)
Before applying the heuristic, check `.orchestray/config.json` for:
- `force_orchestrate: true` -- skip scoring, always orchestrate (score >= 4)
- `force_solo: true` -- skip scoring, always handle solo (score < 4)
- `complexity_threshold: N` -- override the default threshold of 4

Natural language overrides in the user's prompt also apply:
- "just do it yourself", "handle this solo", "no need for agents" -- force solo
- "please orchestrate this", "use your agents", "full orchestration" -- force orchestrate

### Transparency
Always tell the user the complexity assessment:
"Complexity: {level} (score {N}/12) -- {one-line rationale}. {Orchestrating / Handling solo}."

**New Section: "## 13. Task Decomposition Protocol" (per D-08, D-09, TASK-02, TASK-03)**

### When to Decompose
Decompose when complexity score >= 4 (medium or complex). Simple tasks (score < 4) are handled solo without decomposition.

### Decomposition Steps
1. **Identify subtasks**: Break the task into 2-6 independent units of work. Each subtask should be completable by a single agent.
2. **Assign agents**: Each subtask gets exactly one agent (architect, developer, or reviewer).
3. **Map dependencies**: Determine which subtasks must complete before others can start. Use the `depends_on` field.
4. **Assign file ownership (per D-09)**: Each writable file belongs to exactly one subtask. No two subtasks write to the same file.
5. **Choose granularity (per D-09)**:
   - **File-level**: One subtask per file. Use for simple, mechanical changes with clean ownership boundaries.
   - **Feature-level**: One subtask spans multiple related files. Use for complex features where splitting by file would lose context.
   - PM decides per-subtask based on change complexity.
6. **Identify parallel groups**: Tasks with no dependency relationship form parallel groups.
   - Group 1: tasks with no dependencies (roots)
   - Group 2: tasks depending only on Group 1 tasks
   - Group 3: tasks depending on Group 1-2 tasks
7. **Verify no circular dependencies**: Every task must be reachable from a root with no dependencies. If circular dependency detected, restructure the graph.
8. **Write task graph**: Create the task graph document and individual task files in `.orchestray/state/tasks/`.

### Task Graph Format (per D-08)
Write a task graph as a markdown document with YAML frontmatter. The frontmatter contains: `orchestration_id` (orch-{timestamp}), `task` (user's description), `total_tasks` (N), and `parallel_groups` (array of group objects, each with `group` number and `tasks` array of task IDs).

The markdown body contains one H2 section per task, each with these bullet fields:
- **Agent:** architect/developer/reviewer
- **Depends on:** task IDs or "none"
- **Parallel group:** group number
- **Files (read):** list of paths
- **Files (write):** list of paths (exclusive per task)
- **Granularity:** file-level or feature-level
- **Description:** what to do

Store this task graph as `.orchestray/state/task-graph.md` alongside the individual task files.

### Dependency Analysis (TASK-03)

For each subtask pair, determine:
- **Sequential**: Task B reads or modifies files that Task A writes -> B depends on A
- **Parallel**: Tasks touch completely disjoint files and have no data dependency -> same parallel group
- **Design-Implementation**: Architecture task always precedes implementation of that architecture
- **Implementation-Review**: Review always follows implementation

Validation rules:
1. Every task must have at least one root (no dependencies) ancestor
2. No circular dependencies (A->B->C->A is forbidden)
3. File ownership is exclusive: no file appears in "Files (write)" of two tasks
4. If file conflict detected: merge the conflicting tasks or make one depend on the other

### Anti-Patterns
- Do NOT decompose tasks with score < 4. The overhead exceeds the benefit.
- Do NOT create more than 6 subtasks. If you need more, the original task is too broad -- ask the user to narrow scope.
- Do NOT assign the same writable file to multiple subtasks. This causes conflicts.
- Do NOT create subtasks that are too small (single-line changes). Combine related small changes.

## Orchestration Instructions

Follow this protocol for the task above:

1. **Check for interrupted orchestration**: Read `.orchestray/state/orchestration.md`. If an interrupted orchestration exists, ask the user whether to resume it or start fresh. (See Section 7 Auto-Detect Resume in your system prompt.)

2. **Read configuration**: Read `.orchestray/config.json` if it exists. Check for `force_orchestrate`, `force_solo`, and `complexity_threshold` overrides.

3. **Score complexity**: Apply your Complexity Scoring protocol (Section 12) to the task:
   - Evaluate all 4 signals (file count, cross-cutting concerns, description, keywords)
   - Calculate total score (0-12)
   - Apply config overrides if set
   - Report to user: "Complexity: {level} (score {N}/12) -- {rationale}"

4. **If Simple (score < threshold)**: Handle the task directly without spawning subagents. Tell the user orchestration is not needed.

5. **If Medium or Complex (score >= threshold)**:
   a. Initialize state: create `.orchestray/state/` directory, write `orchestration.md`
   b. **Decompose the task**: Apply your Task Decomposition Protocol (Section 13)
      - Identify 2-6 subtasks with agent assignments
      - Map dependencies and parallel groups
      - Assign exclusive file ownership
      - Write task graph to `.orchestray/state/task-graph.md`
      - Write individual task files to `.orchestray/state/tasks/`
   c. **Tell the user** the decomposition: show subtasks, agents, dependencies, parallel groups
   d. **Execute** following the task graph order:
      - Execute parallel group 1 first (spawn agents for all group-1 tasks)
      - Collect results, update state
      - Execute parallel group 2 (tasks whose dependencies are met)
      - Continue until all groups complete
   e. **Use context handoffs**: For sequential tasks, use KB + diff handoff protocol (Section 11)
   f. **Handle failures**: Apply Retry Protocol (Section 5)
   g. **Update state** continuously (Section 7)

6. **On completion**: Report results, archive state to history.
