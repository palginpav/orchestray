The section MUST contain these subsections:

### KB Directory Structure
Document the `.orchestray/kb/` layout:
- `.orchestray/kb/index.json` - Fast lookup index with TTL metadata
- `.orchestray/kb/facts/{slug}.md` - Codebase facts
- `.orchestray/kb/decisions/{slug}.md` - Architectural decisions
- `.orchestray/kb/artifacts/{slug}.md` - Work products (designs, review notes)

### Writing to KB
Instructions for the PM (and to relay to agents):
1. Before writing, read `.orchestray/kb/index.json` and check for existing entries on the same topic (D-04 deduplication)
2. If duplicate found: update the existing file and update `updated_at` in index
3. If new: choose category (facts/decisions/artifacts), create slug (lowercase, hyphens), write markdown detail file
4. Update index.json with entry containing these exact fields:
   - `id`: "{category}-{slug}"
   - `category`: "fact" | "decision" | "artifact"
   - `topic`: human-readable topic description
   - `source_agent`: agent name that wrote it
   - `created_at`: ISO timestamp
   - `updated_at`: ISO timestamp
   - `ttl_days`: 14 for facts, 30 for decisions, 7 for artifacts (D-03)
   - `stale`: false (set to true when TTL expires)
   - `file`: relative path from kb/ directory
   - `summary`: one-line description, 50 tokens max
5. Keep detail files under 500 tokens each
6. Include version field at top level of index.json: `"version": 1`

### Reading from KB
Instructions for selective reading:
1. NEVER read the entire KB. Only read specific named files relevant to the current task.
2. On every KB read, check `created_at + ttl_days` against current date for each entry read
3. If TTL expired: set `stale: true` in index, still return the entry but flag it as "[STALE]" in output
4. When delegating to agents, tell them EXACTLY which KB files to read by path

### Instructing Agents to Write KB
Template for delegation prompts:
"After completing your task, write your key findings to the knowledge base:
- Write to `.orchestray/kb/{category}/{slug}.md`
- Update `.orchestray/kb/index.json` adding your entry
- Keep detail files under 500 tokens
- Include: what you found, why it matters, what the next agent should know"

### KB Initialization
When `.orchestray/kb/` does not exist:
1. Create the directory structure: `kb/`, `kb/facts/`, `kb/decisions/`, `kb/artifacts/`
2. Create `kb/index.json` with `{"version": 1, "entries": []}`
3. Do NOT pre-populate with project facts -- let agents discover and register naturally

### What Agents Register (per D-02)
All agents register everything they discover:
- **Facts**: codebase structure, file purposes, dependency relationships, API contracts
- **Decisions**: architectural choices made during design, tradeoff analysis
- **Artifacts**: design documents, review notes, implementation plans

The section MUST contain:

### Handoff Flow
Document the KB + diff handoff pattern for sequential agents:

1. PM spawns Agent A with task + instruction to write discoveries to KB
2. Agent A writes findings to `.orchestray/kb/{category}/{slug}.md` and updates index
3. PM prepares handoff for Agent B by:
   a. Identifying which KB entries Agent A created or updated (check index.json for entries with Agent A as source_agent and recent updated_at)
   b. Running `git diff` to capture Agent A's code changes
   c. Composing Agent B's delegation prompt with:
      - The task description for Agent B
      - "Read the following KB entries for context: [list specific file paths]"
      - The git diff output (or summary if diff is large)
4. Agent B reads specified KB entries, understands changes via diff, proceeds with its task
5. Agent B writes its own discoveries to KB

### Handoff Delegation Template
Provide a concrete template the PM uses when spawning a sequential agent:

```
[Task description for Agent B]

## Context from Previous Agent
The {previous_agent} completed {previous_task}. Key context:

### KB Entries to Read
- `.orchestray/kb/{category}/{slug-1}.md` - {summary from index}
- `.orchestray/kb/{category}/{slug-2}.md` - {summary from index}

### Code Changes
{git diff output or summary}

Use the KB entries and code changes above to understand the current state before proceeding.
```

### Anti-Patterns
- NEVER dump all KB entries to the next agent -- only entries from the previous agent's work
- NEVER skip the diff -- it shows what actually changed vs what was planned
- NEVER create a separate "handoff document" -- KB + diff IS the handoff (D-10)
- Keep diff output manageable: if >200 lines, summarize the key changes instead of including full diff
