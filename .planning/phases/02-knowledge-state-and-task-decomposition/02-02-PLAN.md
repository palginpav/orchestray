The REPLACEMENT section MUST contain these subsections:

### State Directory Structure (per D-07)
Document the directory layout using a text tree:
- `.orchestray/state/orchestration.md` -- Current orchestration metadata (YAML frontmatter + markdown body)
- `.orchestray/state/tasks/` -- One file per decomposed task: `01-{slug}.md`, `02-{slug}.md`
- `.orchestray/state/agents/` -- One file per agent run: `{agent}-run-{n}.md`

### Orchestration Metadata Format
Document `state/orchestration.md` format. YAML frontmatter fields: `id` (orch-{timestamp}), `task` (user description), `status` (in_progress/completed/failed/interrupted), `started_at` (ISO), `complexity_score` (1-12), `complexity_level` (simple/medium/complex), `delegation_pattern` (sequential/parallel/selective), `total_tasks` (N), `completed_tasks` (N), `current_phase` (assessment/decomposition/delegation/implementation/review/complete). Markdown body: Progress checklist with [x] completed and [ ] pending items, plus Decisions Made section.

### Task File Format
Document `state/tasks/{NN}-{slug}.md` format. YAML frontmatter fields: `id` (task-{NN}), `title`, `status` (pending/in_progress/completed/failed), `assigned_to` (architect/developer/reviewer), `depends_on` (array), `parallel_group` (N or null), `files_owned` (array), `files_read` (array), `started_at` (ISO or null), `completed_at` (ISO or null). Markdown body: Assignment section (PM's delegation prompt) and Result section (agent's result summary).

### Agent Run File Format
Document `state/agents/{agent}-run-{n}.md` format. YAML frontmatter fields: `agent` (architect/developer/reviewer), `task_id` (task-{NN}), `status` (running/completed/failed), `started_at` (ISO), `completed_at` (ISO or null). Markdown body: Result section with structured result from agent.

### Continuous Saving Protocol (per D-05)
State is written on EVERY change, not at boundaries:
1. **Orchestration start**: Create `state/orchestration.md` with status "in_progress"
2. **Task decomposition complete**: Create all `state/tasks/*.md` files with status "pending"
3. **Agent spawn**: Create `state/agents/{agent}-run-{n}.md` with status "running", update task status to "in_progress"
4. **Agent complete**: Update agent run file with result, update task file with status "completed" and result, update orchestration.md completed_tasks count and progress checklist
5. **Agent failure**: Update agent run file with failure, update task file status to "failed"
6. **Orchestration complete**: Update orchestration.md status to "completed", move to `.orchestray/history/{timestamp}-orchestration/` (entire state directory)

### Auto-Detect Resume (per D-06)
When starting ANY orchestration (including via `/orchestray:run`):
1. Check if `.orchestray/state/orchestration.md` exists
2. If exists AND status is "in_progress" or "interrupted": Read all task files to determine what completed and what remains. Tell the user: "Found interrupted orchestration: {task}. {N}/{total} tasks completed. Resume? (yes/no)". If yes: continue from next incomplete task. If no: archive old state to history and start fresh.
3. If not exists or status is "completed": proceed normally

### Backward Compatibility
Keep updating `.orchestray/current-task.json` alongside the new state directory for backward compatibility with Phase 1 status skill. The state directory is the source of truth; current-task.json is a convenience mirror.

### State Recovery
If `orchestration.md` is corrupted but task files exist: Regenerate orchestration.md from task file frontmatter. Count completed/pending tasks to reconstruct progress. Set status to "interrupted" to trigger resume flow.

IMPORTANT: When writing the YAML format examples in the PM prompt, use indented code blocks (4 spaces before the triple-backtick fence) so that any `---` inside the example does not appear at column 0. This prevents frontmatter parser confusion in downstream tools.
