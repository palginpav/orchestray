**SECTION 17: Dynamic Agent Spawning Protocol**

Content must include:

1. **When to Spawn Dynamic Agents** -- Decision criteria during task decomposition (Section 13) or re-planning (Section 16):
   - Subtask requires domain expertise not covered by architect/developer/reviewer (e.g., database migration, security audit, performance profiling, documentation, testing)
   - Subtask has unique tool restrictions different from core agents
   - Subtask benefits from a highly focused system prompt
   - Per D-03: No hard cap on dynamic agents -- PM decides based on task needs, token budget is the natural limit
   - Dynamic agents should be RARE. Most tasks fit architect/developer/reviewer. Per research: 3-5 agent sweet spot, dynamic agents add overhead.

2. **Agent Definition Generation** -- PM writes a temporary agent definition file to `agents/` directory. The file must follow Claude Code's standard agent `.md` format (YAML frontmatter + markdown body). Template:

   ```yaml
   ---
   name: {task-specific-name}
   description: {one-line description of specialization}
   tools: {appropriate tool subset -- see tool access patterns below}
   model: inherit
   maxTurns: 30
   color: cyan
   ---
   ```

   Followed by a focused system prompt (100-200 lines MAX -- see Pitfall 3 in research). The prompt MUST include:
   - Task-specific instructions and domain knowledge
   - The project's standard output format (Section 6 JSON+markdown)
   - KB protocol instruction (Section 10 -- write findings to KB)
   - Explicit scope boundaries (what to do and what NOT to do)

3. **Tool Access Patterns** (per D-04, at Claude's discretion):
   - Read-only specialists (analysis, audit, profiling): `Read, Glob, Grep, Bash`
   - Write specialists (implementation, migration, documentation): `Read, Glob, Grep, Bash, Write, Edit`
   - Choose the minimal tool set that enables the task

4. **Lifecycle** -- Ephemeral per D-04:
   a. PM creates agent definition file at `agents/{name}.md` before spawning
   b. PM spawns the agent using `Agent("{name}")` -- Claude Code resolves the name to the file
   c. Agent executes, returns structured result per Section 6
   d. PM processes result per Section 4
   e. PM deletes the agent definition file after processing the result
   f. Log `dynamic_agent_spawn` event to audit trail with: timestamp, type, orchestration_id, agent_name, task_id
   g. Log `dynamic_agent_cleanup` event after deletion

5. **Note on Agent() Tool** -- The PM frontmatter lists `tools: Agent(architect, developer, reviewer)` which documents the core agents. Claude Code's Agent() tool resolves any agent name to a matching `.md` file in the `agents/` directory. The PM is NOT restricted to only the listed names. To spawn a dynamic agent, simply use `Agent("{dynamic-name}")` after creating the definition file.

**SECTION 18: Verify-Fix Loop Protocol**

This section REPLACES the old Section 5 (Retry Protocol). The old Section 5 allowed one blind retry. Section 18 implements structured multi-round quality loops.

Content must include:

1. **When to Enter Verify-Fix Loop** -- Triggered when:
   - Reviewer returns `status: "failure"` with `issues` containing `severity: "error"` items
   - Only error-severity issues trigger the loop (warnings and info do NOT)
   - The failure is an implementation bug, NOT a design flaw (design flaws route to Section 16 re-planning)

2. **Loop Mechanics** -- For each round:
   a. **Extract fix instructions from reviewer:** Parse the reviewer's structured result. Filter `issues` array to `severity: "error"` only. Each issue's `description` field contains file path, line reference, and suggested fix (per reviewer protocol). This becomes the fix instructions.
   b. **Build developer fix prompt:** Include:
      - Original task context (abbreviated -- task title, key files, constraints)
      - Specific issues to fix (from reviewer, verbatim error-severity issues)
      - Files that need changes (extracted from issue descriptions)
      - Cumulative fix history: ALL issues fixed in previous rounds with explicit instruction "These issues were fixed in previous rounds and MUST remain fixed. Do not regress them."
      - Attempt counter: "This is attempt {N} of {max_rounds}" (where max_rounds comes from `verify_fix_max_rounds` config, default 3 per D-05)
      - Instruction: "Fix ONLY the listed issues. Do not refactor, add features, or make unrelated changes."
   c. **Spawn developer** with the fix prompt
   d. **Spawn reviewer** to re-validate the developer's fixes
   e. **Evaluate reviewer result:**
      - If `status: "success"` (no error-severity issues) -> EXIT loop, proceed normally
      - If errors remain AND round < max_rounds -> CONTINUE loop (go to step a)
      - If errors remain AND round >= max_rounds -> ESCALATE to user (step 3)
      - If developer reports the fix is impossible or requires design change -> TRIGGER re-plan (Section 16)
   f. **State tracking:** Update `.orchestray/state/tasks/{task}.md` with verify_fix block:
      ```yaml
      verify_fix:
        rounds_completed: {N}
        max_rounds: {max}
        round_history:
          - round: 1
            reviewer_issues: {count}
            developer_fixed: {count}
            remaining: {count}
        status: in_progress | resolved | escalated | design_rejected
      ```
   g. **Audit logging:** For each round, append to `.orchestray/audit/events.jsonl`:
      - Round start: `{"type": "verify_fix_start", "orchestration_id": "...", "task_id": "...", "round": N, "error_count": N}`
      - Round pass: `{"type": "verify_fix_pass", "orchestration_id": "...", "task_id": "...", "round": N, "rounds_total": N}`
      - Round fail (cap reached): `{"type": "verify_fix_fail", "orchestration_id": "...", "task_id": "...", "round": N, "remaining_errors": N}`

3. **User Escalation** (per D-06) -- When verify_fix_max_rounds is reached:
   ```
   Verify-fix loop reached maximum rounds ({N}/{max}).
   
   Remaining issues:
   - {list of unresolved error-severity issues from last reviewer pass, verbatim}
   
   Options:
   1. Accept current implementation with known issues (I'll document the warnings)
   2. Provide guidance for another attempt (describe what to try differently)
   ```
   Log escalation event: `{"type": "escalation", "orchestration_id": "...", "task_id": "...", "reason": "verify_fix_cap", "remaining_errors": N}`
   If user chooses option 1: set verify_fix status to "escalated", proceed
   If user chooses option 2: reset round counter with user's guidance as additional context, continue loop

4. **Regression Prevention** -- The cumulative fix history in each developer prompt is the primary anti-regression mechanism. Include ALL previously fixed issues with the instruction "MUST remain fixed." If round N has same or more errors than round N-1, flag this as a potential oscillation pattern in the audit log and consider escalating early.
