**Section 16: Adaptive Re-Planning Protocol**

Content must include all of the following subsections:

1. **When to Re-Plan** -- Trigger evaluation. After Section 4 processes each agent result, evaluate these signals (per D-01, all agent feedback triggers consideration):
   - Approach invalidation: agent reports the chosen approach is fundamentally flawed
   - Scope expansion: agent discovers the task is significantly larger than estimated
   - Dependency discovery: agent found a dependency not captured in the original graph
   - Completed work invalidation: new information means already-completed work is wrong (per D-02, no completed-work lock-in)
   - Reviewer design rejection: reviewer identifies a design flaw (not implementation bug -- implementation bugs route to verify-fix loop in Section 18)
   - If NONE of these signals present, proceed with current graph (no re-plan)

2. **Re-Plan Execution** -- Steps:
   a. Read current task graph from `.orchestray/state/task-graph.md`
   b. Read all completed task results from `.orchestray/state/tasks/*.md`
   c. Evaluate which completed work is still valid given the new information (per D-02, can revisit completed work)
   d. Generate a new task graph by re-running Section 13 decomposition with updated context including the new information
   e. Log re-plan event to audit trail `.orchestray/audit/events.jsonl` with JSON fields: timestamp, type "replan", orchestration_id, reason (one of: approach_invalidation, scope_expansion, dependency_discovery, completed_work_invalidation, design_rejection), old_task_count, new_task_count, tasks_invalidated (array of task IDs)
   f. Register re-plan decision in KB (`.orchestray/kb/decisions/replan-{timestamp}.md`) with reason and summary of what changed
   g. Update `.orchestray/state/orchestration.md` with incremented `replan_count` field
   h. Update state files for any invalidated tasks: set status to `invalidated` with reason

3. **Anti-Thrashing Safeguard** -- Track re-plan count per orchestration. Read `replan_budget` from `.orchestray/config.json` (default 3). After reaching the budget, PM MUST escalate to user instead of re-planning again. Escalation message format:
   ```
   Re-planning budget reached ({N}/{budget}).
   
   Current situation: [summary of what's happening]
   
   Options:
   1. Increase re-plan budget and continue (I'll try a different approach)
   2. Provide guidance on how to proceed
   3. Abort this orchestration
   ```

4. **Distinguishing Re-Plan from Verify-Fix** -- Clear decision tree:
   - Implementation bug (reviewer found code errors) -> Section 18 (verify-fix loop)
   - Design flaw (approach is wrong) -> Section 16 (re-plan)
   - Scope change (task is bigger/different than expected) -> Section 16 (re-plan)
   - Agent failure (crash, timeout) -> Section 5 (retry, then re-plan if retry fails)

Also make these extensions to existing sections:

**Section 0 (Auto-Trigger Protocol), Medium+ Task Path step 4:** After "After each agent completes: display running costs (Section 15, step 2)", add: "evaluate for re-plan signals (Section 16)."

Most results will NOT trigger re-planning -- only structural signals warrant graph restructuring. Implementation bugs are handled by the verify-fix loop (Section 18), not re-planning."

**Section 7 (State Persistence Protocol):** Add a new trigger point #7: "Re-plan executed: Update orchestration.md with incremented replan_count, update invalidated task files with status invalidated, write new task-graph.md."

**Section 9 (Anti-Patterns):** Add rule #9: "Never re-plan on implementation bugs. If the reviewer found code errors (missing null check, wrong return type, test failure), that is a verify-fix loop (Section 18), not a re-planning trigger. Re-planning is for structural problems: wrong approach, scope change, missing dependencies. Misusing re-plan for bug fixes wastes the re-plan budget and delays resolution."

**Section 13 (Task Decomposition Protocol):** Add a subsection "### Re-Planning Entry Point" at the end: "When invoked from Section 16 (re-planning), the decomposition steps are identical but with additional context: (1) The original task graph is available in .orchestray/state/task-graph.md. (2) Completed task results are available in .orchestray/state/tasks/*.md. (3) Some completed tasks may be marked as invalidated -- their work should be redone. (4) The re-plan reason provides context about what changed. Use all of this to produce a revised task graph that accounts for the new information."
